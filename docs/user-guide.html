What It Is (and Is Not)

foundry packages the master workflow from the companion research guide (Knowledge Base/research/research_ai-engineering-guide_2026-07-04.md) as a working plugin. Its bet: front-load the thinking — grilling, research, prototype, PRD, board — and autonomous execution becomes safe and good. 'AFK execution is a consequence of setup, not a goal.'

  • It is a process harness: skills that orchestrate each phase, subagents that keep contexts clean, hooks that make the guardrails deterministic.
  • It is not a replacement for judgement: the PRD approval, the QA walkthrough, and the comprehension quiz are human gates by design — the plugin schedules them, it never performs them.
  • Every skill works standalone. You can adopt /foundry:grill or the tdd discipline without ever running /foundry:ship. Nothing owns your process.

Install

foundry is two layers, split cleanly by directory: skills/ is the portable pipeline logic (open Agent Skills standard — works on any compatible harness); hooks/, agents/, and .claude-plugin/ are Claude Code-only guardrails and won't run elsewhere. Everywhere else, the pipeline still works — the skill text itself instructs the agent on the rules; you just lose deterministic enforcement.

MimoCode, MiniMax, others

Not verified against a real install — neither runtime was present on the machine this plugin was built on. If the harness reads SKILL.md + YAML frontmatter from a scanned directory, copying skills/<name>/ in directly should work; unrecognised Claude-Code-only frontmatter keys (disable-model-invocation, context: fork, agent:) should be safely ignored. Treat this as a starting point, not a confirmed recipe.

MCP servers

The bundled .mcp.json registers the hosted GitHub and Linear servers. Authenticate on first use via /mcp. Using only one tracker? Delete the other entry. Every new server you add later should pass /foundry:security-review first.

Hooks activate when the plugin is enabled and apply to the session. Read hooks/scripts/*.sh before installing — they are short, plain bash, and they run with your permissions. That review habit applies to any plugin, including this one.

Quickstart (Five Minutes)

cd your-repo
claude

> /foundry:ship add CSV export to the reports page
  • foundry creates .foundry/state.json, records your idea in intent.md, and starts grilling you — one question at a time. Answer concretely; say 'parked' to defer a question.
  • It then offers Research/Prototype only if warranted, drafts the PRD, and waits for your approval.
  • On approval it creates vertical-slice tickets with blocking links, then implements them one at a time under TDD, opening a PR per ticket.
  • Finally it hands you a QA plan, a literate diff, and a five-question quiz. Findings become new tickets; the loop repeats until the board is empty.

At any point, plain /foundry:ship reports where you are and the single next command.

The Pipeline

PHASE 1Idea/foundry:grillgate: decisions resolved
PHASE 2 · conditionalResearchresearchresearch.md + expiry
PHASE 3 · conditionalPrototypeprototypehuman picks winner
PHASE 4PRD/foundry:prdHUMAN APPROVES
PHASE 5Board/foundry:boardvertical slices
PHASE 6Execute/foundry:implementcheck passes
PHASE 7QA/foundry:review · /foundry:qaHUMAN + QUIZ

Phases 5–7 loop until the board is empty and all gates pass. Dashed phases are skipped when they add no value — foundry must justify offering them in one line. The one-sentence rule: a change you could describe as a single-sentence diff skips straight to implementation.

Worked Example

Shipping 'CSV export on the reports page' end to end

1 · Idea. /foundry:ship add CSV export to the reports page. Grilling asks: which report views? all rows or the filtered set? who may export? filename convention? what happens over 100k rows? Answers land in intent.md; 'column localisation' gets parked.

2 · Research — skipped. No unfamiliar external (CSV is stdlib territory). foundry says so in one line and moves on.

3 · Prototype — offered. Streaming download v background-job-with-email is a real architecture question. Two spikes in a worktree; you pick streaming; the winner is committed as prototype: streaming csv spike.

4 · PRD. Draft describes the button, the filtered-set semantics, the failure banner over 100k rows, and five pass/fail ACs (e.g. 'exported file opens in Excel with correct headers — check: npm run test:e2e -- csv-export'). You approve; it lands as the parent issue.

5 · Board. Three vertical slices: export endpoint with streaming + tests → button + download flow + e2e (blocked by #1) → permissions + audit log (unblocked). You sanity-check the table.

6 · Execute. Ticket 1: explorer maps the report API; planner emits four TDD steps; implementation runs red–green–refactor; the Stop hook blocks an early 'done' because the e2e check still fails; iteration 2 fixes it; PR opens. Tickets 2–3 run in parallel worktrees.

7 · QA. Review (fresh context) flags a should-fix: the endpoint duplicates an existing serialiser — ticketed. The QA plan walks you through export-with-filters and the >100k failure banner; the quiz asks why streaming beat the job queue and where the row cap lives. You pass; the duplicate-serialiser ticket loops through 5→6→7; board empties; the parent issue closes.

Guardrails (What the Hooks Enforce)

HookFires onEnforces
protect-branchesevery Bash callblocks pushes to main/master, force-pushes, and direct commits/merges on main
deny-destructiveevery Bash callblocks rm -rf on root/home, curl-pipe-to-shell, hard resets to remote main
formatevery Edit/Writebest-effort prettier/ruff/gofmt/rustfmt on the touched file; never blocks
test-gateturn end (Stop)while a ticket is active, the turn cannot end until its acceptance check passes (harness overrides after 8 consecutive blocks)
board-statesession startinjects phase/ticket/gates; warns at iteration count 2 of 3
research-expirysession startwarns when research.md is past its expiry (research rots)
The 3-iteration rule. After three consecutive agent-only attempts at the same failing check, foundry stops and demands human review — unreviewed iterative 'improvement' measurably increases vulnerabilities (+37.6% criticals over five iterations in controlled testing). Do not raise this cap.

Hooks are deterministic where CLAUDE.md-style instructions are advisory — that is why the guarantees live here. To bypass a gate deliberately (rare), set gate_active: false in .foundry/state.json and say why in the session.

Parallel and AFK Operation

Parallel tickets

Ask /foundry:implement to parallelise: it fans foundry-implementer subagents across genuinely unblocked tickets, each in its own git worktree, so edits never collide. Add a .worktreeinclude file (gitignore syntax) listing untracked essentials such as .env so new worktrees get them automatically.

AFK (the Ralph loop)

# emitted by /foundry:implement on request — review before running
while next=$(gh issue list --label ready --limit 1 -q '.[0].number' --json number); do
  [ -z "$next" ] && break
  claude -p "/foundry:implement #$next" \
    --allowedTools "Read,Edit,Write,Bash(npm test:*),Bash(git commit:*),Bash(gh pr create:*)"
done
  • Fresh context per ticket; the board is the memory — that is what makes walking away safe.
  • Only after phases 1–5 are done well: AFK quality is a consequence of setup.
  • Set a budget first. Heavy 2026 agent use runs to hundreds of dollars a day; foundry asks for a per-sprint cap before starting.
  • QA and the quiz still happen when you return. AFK moves the execution, never the gates.

Troubleshooting and FAQ

The skills don't appear after install

Run /plugin → check foundry is installed and enabled, then restart the session. For standalone copies, confirm the folder is .claude/skills/<name>/SKILL.md (the folder name matters) and keep the *-FORMAT.md companions next to their SKILL.md — they are referenced by relative path.

The turn keeps getting blocked at the end

That is the test-gate doing its job: .foundry/state.json has gate_active: true and the ticket's check_command is failing — the last 20 lines of its output are shown. Fix the failure, or if the check itself is wrong, correct it on the ticket and in state.json. Genuinely blocked? Set gate_active: false and say why; the harness also overrides after 8 consecutive blocks.

foundry keeps offering phases I don't want

Say 'skip research' / 'skip prototype' — they are conditional by design and your call is final. If it happens repeatedly, the intent is probably under-specified: run /foundry:grill once more and record the decision, and it will stop asking.

Can I use it without GitHub or Linear?

Yes — tell /foundry:board to use local files; it will keep the board as board/ markdown tickets with Blocked by: lines. You lose true blocking semantics and PR integration, but the pipeline works. Remove the unused servers from .mcp.json.

How do I hand work to a colleague (or to tomorrow's session)?

/foundry:handoff. It writes handoff.md — decisions with their why, unresolved items, gotchas, next actions — and syncs state.json. A fresh session picks up with /foundry:ship and needs nothing from the old conversation.

The agent claims something works but it doesn't

Three levers, in order: (1) make the ticket's acceptance check executable — an agent cannot argue with an exit code; (2) run foundry-verifier, which is adversarial by contract and reports UNVERIFIABLE rather than PASS when it could not exercise a criterion; (3) file the case with /foundry:evals so the regression is caught automatically next time. And the permanent rule: verification is ultimately yours — the QA walkthrough is not optional.

Where do I change what the hooks block?

hooks/scripts/*.sh — short, commented bash. Edit the patterns, then restart the session. Keep protect-branches and the 3-iteration rule unless you have a very good reason; they exist because the failure modes they stop are silent.
